# Тестовое задание

Представим себе, что мы продуктовая компания, пишем ПО для каких-то предприятий.
В результате анализа потребностей клиентов компания пришла к выводу, что у предприятий существует потребность в качественном
обучении сотрудников. У компании уже есть некоторое количество походящих учебных курсов и есть научно-исследовательский потенциал для создания новых курсов.

ЗАДАЧА:
Нужно написать сервис, предоставляющий API для доступа сотрудников к учебным курсам.


 Требования к выполнению
 - Язык php >= 7.3 или Go 1.11.*
 - Желательно MVC
 - Любые доп. технологии (фреймворки, библиотеки) если нужно.
 - Хранилище данных любое, хоть mysql, хоть json в файле, главное чтобы были какие-то первоначальные данные. Должна быть возможность быстро перейти с одного хранилища на другое без переписывания всего приложения (буква D в SOLID).
 - доступ к курсам предоставляется в виде одноразовых ссылок. После прохождения курса ссылка недействительна. Одна ссылка - доступ к единственному курсу на одно прохождение.
 - Фронт никакой не писать, только бек. Предполагаем, что фронт будет писаться кем-то другим.
 - Врохождение курса - полностью фронтовая задача. Все что мы знаем о курсе: курс написан на html+js и воспроизводится полностью в браузере сотрудника предприятия.
      В процессе прохождения курса с фронта приходят запросы с инфой о процессе прохождения. Для нас процесс прохождения - проценты от 10 до 90. 
      В конце прохождения (100%) курс (фронт) отправляет на бек http запрос, означающий, что курс пройден, значит ссылку делаем недействительной. В запросе
 - Учебный курс - сущность со свойствами id, name. Все остальные детали мы опустим.
 
 Методы, которые должны быть у сервиса:
 - Получене курса по одноразовой ссылке. Принимает имя сотрудника и название его предприятия.
 - Фиксация состояния процесса прохождения курса сотрудником.
 - Фиксация окончания прохождения курса. Этот метод можно совместить с прдыдущим, а можно сделать отдельным.
 
 
 Требовния к коду:
 - В контроллерах (или иных приинимающих решения классах, если будет не MVC) бизнес-логику не писать. Соблюдать конецпцию тонких контроллеров.
 - Доступ к курсам и остальные операции (фиксация состояния прохождение, завершение курса сотрудником) должны быть реализованы через сервисный слой.
 - Чистый код, у методов указаны возвращаемые значения, у параметров указа тип (речь не о phpdoc, а о сигнатурах методов)
 - Код с соблюдением psr12
 
 Будет плюсом:
  - Unit тесты
  - Запуск сервиса через docker-compose up
  - Описание в виде текста и/или диаграмм
  
 Термины:
 - Сервисный слой - по сути это просто классы типа SomeService, которые отвечают за работу с какими-то сущностями. Ничего не знают о том, кто их использует, знают все о сущностях, с которыми через них работают.
